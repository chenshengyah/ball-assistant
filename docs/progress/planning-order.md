# MVP 前期规划顺序

这份文档用于指导当前 MVP 的前期规划顺序，核心原则是先按主链路规划，而不是先把“角色创建”单独拆成一个大专题。角色 / 身份需要规划，但默认作为创建活动链路中的一个关键节点处理。

当前 P0 规划主线固定为：

`完善信息 -> 确认发布身份 -> 俱乐部注册 -> 创建活动（必要时补充场馆与场地） -> 活动详情 -> 报名/取消报名 -> 我的活动回收`

相关文档：

- `docs/global/mvp-overview.md`：目标和范围来源
- `docs/global/user-flow.md`：主链路基础
- `docs/progress/mvp-progress.md`：记录当前做到哪一步

## 规划原则

- 先主链路：优先回答“用户如何一路走到成功发布和报名”，不要先拆孤立模块。
- 后支撑模块：角色边界、字段口径、接口契约都服务于主链路，不单独先做成大系统。
- 最后补全收口：主链路稳定后，再冻结页面职责、首批开发清单和补充文档。

## 推荐规划顺序

### 阶段 1：锁定 P0 主链路

- 目标：先明确从首次进入到成功报名的完整路径，确保 MVP 的核心目标是同一条线。
- 为什么先做这个：如果主链路不先定，后面的角色、页面和字段规划都会散。
- 需要回答的问题：
  - 用户第一次进入时先去哪里？
  - 用户怎么从建档走到发布活动？
  - 其他用户怎么从活动列表走到报名成功？
  - 整条链路里哪些页面是 P0 必须有的？
- 产出物：
  - 一条统一的 P0 主链路说明
  - 主链路节点清单：完善信息、发布身份、俱乐部注册、创建活动、活动详情、报名/取消报名、我的活动
- 进入下一阶段的条件：团队已经能用一句话说清楚“用户如何完成一次发布和报名”。

### 阶段 2：拆主链路中的关键分叉

- 目标：把主链路中会影响页面和规则设计的分叉场景提前拆清楚。
- 为什么先做这个：很多看起来像“角色规划”的问题，本质上是链路分叉问题，不先拆分叉会把身份系统做重。
- 需要回答的问题：
  - 用户未建档时如何拦截和回跳？
  - 用户没有俱乐部身份时如何进入注册并返回？
  - 用户没有合适场馆或场地时，如何在创建页内补充并继续创建？
  - 活动创建成功后去活动详情还是活动列表？
  - 用户报名成功后在哪里看到自己的状态？
- 产出物：
  - 主链路分叉清单
  - 回跳规则说明
  - “角色 / 身份”在主链路里的位置说明
- 进入下一阶段的条件：已明确“有 / 无俱乐部”“有 / 无场馆”“已建档 / 未建档”等关键分叉和承接关系。

### 阶段 3：把流程挂到页面

- 目标：把主链路和分叉逐个挂到具体页面，形成页面地图和页面职责。
- 为什么先做这个：只有页面职责定下来，PRD 和原型才知道该补哪里。
- 需要回答的问题：
  - 每个链路节点由哪个页面负责？
  - 页面入口在哪里？
  - 页面之间如何跳转和回跳？
  - 现在已有页面能承接哪些节点，缺哪些页面？
- 产出物：
  - 页面地图
  - 页面职责清单
  - 现有页面 / 缺失页面对照表
- 进入下一阶段的条件：MVP 页面清单、入口位置、跳转关系和回跳关系都已明确。

### 阶段 4：补齐支撑文档

- 目标：在主链路和页面职责稳定后，再补齐角色边界、字段口径、接口契约和数据字典差异。
- 为什么先做这个：这类文档只有在主链路稳定后补，才不会反复返工。
- 需要回答的问题：
  - 角色边界如何统一表达？
  - 页面字段和全局数据字典是否一致？
  - 当前接口契约能否覆盖页面所需数据？
  - 现有类型命名和文档命名有哪些差异？
- 产出物：
  - 角色边界说明
  - 字段 / 枚举对齐清单
  - 接口契约差异清单
- 进入下一阶段的条件：支撑文档已经能为页面 PRD、原型和后续开发提供统一口径。

### 阶段 5：冻结首批开发顺序

- 目标：把前面的规划结果转成真正能执行的 P0 开发顺序。
- 为什么先做这个：前期规划的价值最终要落到“先开发什么、后开发什么”。
- 需要回答的问题：
  - 哪些页面和能力必须先做，才能跑通主链路？
  - 哪些能力是前置依赖？
  - 哪些内容可以放到 P1，不阻塞 P0？
- 产出物：
  - 首批开发清单
  - P0 / P1 初步划分
  - 每项开发任务依赖的文档说明
- 进入下一阶段的条件：已形成可直接执行的第一批开发顺序，且顺序固定为：
  - 完善信息
  - 创建活动前置能力（发布身份 / 俱乐部注册承接）
  - 创建活动内的场馆 / 场地配置承接
  - 创建活动
  - 活动列表 / 活动详情 / 报名
  - 我的活动 / 我的页面回收

## 每阶段记录模板

后续如果要继续细化某一阶段，统一按这个模板记录：

- 目标
- 为什么先做这个
- 需要回答的问题
- 产出物
- 进入下一阶段的条件

## 使用方式

- 当你在想“现在应该先规划哪部分”时，优先看这份文档，按阶段推进。
- 当你在记录“目前做到哪一步”时，更新 `docs/progress/mvp-progress.md`。
- 当某个阶段产出稳定后，再回填对应的全局文档或页面 PRD，不把方法说明和状态记录混在一起。
