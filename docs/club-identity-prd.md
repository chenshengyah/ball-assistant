# 俱乐部身份与注册产品文档

面向对象：产品、UI 设计、前端开发、后端开发

版本：v1

更新时间：2026-03-30

## 1. 目标

在活动发布场景中，主办方需要区分为：

- 个人发布
- 俱乐部发布

当前阶段重点补齐“俱乐部身份”这条链路，包括：

- 一个账号如何同时拥有个人和俱乐部两种发布身份
- 当用户想以俱乐部身份发布活动，但尚未注册俱乐部时，如何完成注册
- 注册完成后，如何回到活动发布页继续发布

个人发布流程较简单，本篇不展开，重点描述俱乐部方案。

## 2. 核心结论

### 2.1 一个账号是否可以同时有两种身份

结论：可以，且建议这样设计。

原因：

- 登录主体应始终是“用户账号”
- 发布主体可以是“个人”或“俱乐部”
- 同一个用户可能既会自己组局，也会代表俱乐部发活动
- 后续还要支持一个用户管理多个俱乐部，因此不能把“账号”和“俱乐部”绑定成一对一

设计原则：

- 账号是登录身份
- 个人 / 俱乐部是发布身份
- 发布时选择本次使用哪个身份

### 2.2 发布身份的产品规则

创建活动时，“发布身份”字段保留单选。

可选项来源：

- 默认一定有一个“个人”
- 若当前用户是某俱乐部的负责人或管理员，则可额外看到对应俱乐部

示例：

- 个人 · 小陈
- 企鹅羽球俱乐部 · 负责人
- Rally 周末局 · 管理员

权限规则：

- 个人活动：仅创建人本人可管理
- 俱乐部活动：俱乐部 `OWNER / ADMIN` 可管理
- 俱乐部 `MEMBER` 仅可报名，不可代表俱乐部发布活动

## 3. 角色与数据关系

推荐继续沿用当前项目已有思路：

- `User`：登录账号
- `Club`：俱乐部主体
- `ClubMember`：用户和俱乐部的关系
- `Activity.ownerType`：`PERSONAL | CLUB`
- `Activity.ownerId`：指向 `userId` 或 `clubId`

这样可以天然支持：

- 一个账号发布个人活动
- 一个账号代表一个或多个俱乐部发布活动
- 活动列表和详情页统一渲染“发布主体”

## 4. 俱乐部注册入口

## 4.1 入口位置

建议提供两个入口：

- 入口 A：活动创建页选择“发布身份”时，如果没有可用俱乐部身份，展示“注册俱乐部”
- 入口 B：我的页面增加“我的俱乐部 / 注册俱乐部”入口

## 4.2 创建活动页联动

创建活动页中，发布身份区域建议增加以下规则：

- 当用户已有可管理俱乐部时：
  - 下拉项展示“个人 + 俱乐部列表”
- 当用户没有可管理俱乐部时：
  - 默认仅显示“个人 · 我”
  - 在字段下方展示次级入口：`注册俱乐部后可用俱乐部身份发布`

如果用户明确要发俱乐部活动，推荐流程：

1. 用户进入创建活动页
2. 在发布身份区域点击 `注册俱乐部`
3. 完成注册
4. 自动返回创建活动页
5. 自动选中刚注册的俱乐部作为发布身份
6. 保留用户已填写的活动表单内容

这里“保留活动表单内容”很重要，否则会打断发布流程。

## 5. 俱乐部注册页字段

俱乐部注册页建议拆成 3 个区块：基础资料、位置信息、联系方式。

### 5.1 基础资料

- 俱乐部封面
  - 类型：图片上传
  - 数量：1 张
  - 是否必填：是
  - 用途：用于俱乐部主页顶部背景、活动卡片品牌展示
- 俱乐部 Logo
  - 类型：图片上传
  - 数量：1 张
  - 是否必填：是
  - 用途：用于头像、列表、详情页
- 俱乐部名称
  - 类型：单行输入
  - 是否必填：是
  - 规则：2 到 30 字；同名允许与否可由后端后续决定，前端先做非空和长度校验
- 俱乐部品类
  - 类型：单选
  - 是否必填：是
  - 当前版本固定值：`羽毛球`
  - 说明：虽然当前只有羽毛球，也建议按枚举字段存储，方便未来扩展篮球、网球等
- 俱乐部简介
  - 类型：多行输入
  - 是否必填：否
  - 用途：介绍俱乐部定位、组织频率、适合人群、报名规则

### 5.2 位置信息

- 俱乐部地点
  - 类型：腾讯地图 POI 选择
  - 是否必填：是
  - 交互：点击后跳转地图选点或搜索地点
  - 返回并保存以下字段：
    - `poiName`
    - `address`
    - `province`
    - `city`
    - `district`
    - `latitude`
    - `longitude`

说明：

- 俱乐部地点是“俱乐部常驻或默认活动区域”
- 它不等于具体活动场馆
- 活动发布时仍然单独选择实际场馆

### 5.3 联系方式

- 联系人
  - 类型：单行输入
  - 是否必填：是
- 联系电话
  - 类型：手机号输入
  - 是否必填：是
  - 规则：校验手机号格式
- 微信号
  - 类型：单行输入
  - 是否必填：否
  - 说明：如果业务上希望提高触达率，也可改为必填，但 v1 建议先设为选填

## 6. 建议的数据结构

当前项目中的 `Club` 只有 `id / name / status`，建议扩展为：

```ts
type ClubCategory = "BADMINTON";

type ClubStatus = "PENDING" | "ACTIVE" | "INACTIVE";

type Club = {
  id: string;
  name: string;
  logoUrl: string;
  coverUrl: string;
  category: ClubCategory;
  description: string;
  province: string;
  city: string;
  district: string;
  address: string;
  latitude: number;
  longitude: number;
  poiName: string;
  contactName: string;
  contactPhone: string;
  wechatId: string;
  status: ClubStatus;
  createdBy: string;
  createdAt: string;
  updatedAt: string;
};
```

`ClubMember` 结构建议继续保留：

```ts
type ClubRole = "OWNER" | "ADMIN" | "MEMBER";

type ClubMember = {
  id: string;
  clubId: string;
  userId: string;
  role: ClubRole;
};
```

注册成功时，系统自动创建一条成员关系：

- `userId = 当前用户`
- `role = OWNER`

## 7. 状态设计

俱乐部建议至少有以下状态：

- `PENDING`
  - 已提交注册，待审核
- `ACTIVE`
  - 审核通过，可正常展示、可用于发布活动
- `INACTIVE`
  - 停用，不可继续作为发布主体

如果当前阶段不做审核，也建议保留状态字段。

简化方案：

- v1 提交后直接进入 `ACTIVE`
- 后续再接审核流程时，不需要重构数据结构

## 8. 交互流程

### 8.1 首次以俱乐部身份发布

1. 用户进入创建活动页
2. 发现当前只有个人身份
3. 点击 `注册俱乐部`
4. 进入俱乐部注册页
5. 填写封面、Logo、名称、品类、地点、简介、联系人、联系电话、微信号
6. 提交成功
7. 自动创建当前用户为该俱乐部 `OWNER`
8. 返回创建活动页
9. 自动选中新注册俱乐部
10. 用户继续填写并发布活动

### 8.2 已注册俱乐部后发布

1. 用户进入创建活动页
2. 在“发布身份”中选择对应俱乐部
3. 按普通流程完成活动创建

## 9. 页面校验规则

俱乐部注册页建议至少校验：

- 封面必传
- Logo 必传
- 俱乐部名称必填
- 俱乐部品类必填
- 俱乐部地点必填
- 联系人必填
- 联系电话必填且格式正确

建议提示文案：

- `请上传俱乐部封面`
- `请上传俱乐部 Logo`
- `请填写俱乐部名称`
- `请选择俱乐部品类`
- `请选择俱乐部地点`
- `请填写联系人`
- `请填写联系电话`
- `联系电话格式不正确`

## 10. 与活动发布的边界

为了避免概念混淆，需要明确：

- 俱乐部资料中的“俱乐部地点”是俱乐部默认归属地
- 活动中的“场馆”是具体举办场地

两者都需要，但作用不同：

- 俱乐部地点：用于俱乐部主页、搜索筛选、附近俱乐部
- 活动场馆：用于活动报名、导航、场地容量配置

## 11. v1 实现建议

为了先把流程跑通，建议按以下顺序落地：

1. 保留当前“一个账号多发布身份”的设计
2. 新增俱乐部注册页
3. 扩展 `Club` 数据结构
4. 创建活动页补“无俱乐部时去注册”的入口
5. 注册成功后回跳创建页并自动选中俱乐部

当前版本可以先不做：

- 俱乐部审核后台
- 俱乐部主页
- 俱乐部成员邀请
- 一个账号切换默认身份
- 俱乐部认证标识

## 12. 最终建议

基于当前需求，推荐方案是：

- 一个账号可以同时拥有“个人”和“俱乐部”两种发布身份
- 活动创建时通过“发布身份”选择本次使用哪个主体
- 若要用俱乐部身份发布，但当前还没有俱乐部，则先注册俱乐部
- 注册成功后自动回到活动发布页继续流程

这个方案对用户理解最自然，也和当前项目里 `ownerType + ownerId` 的实现方向完全一致，改造成本最低。
