# 活动全链路说明

## 1. 目的

本文统一 MVP 阶段活动从创建到结束的链路口径，重点收敛本轮新增的“角色卡先行、半屏补齐、封面上传、手机号联系口径”。

## 2. 主链路范围

活动全链路覆盖以下节点：

- 进入创建活动页
- 选择发布角色
- 必要时补齐个人或俱乐部资料
- 填写活动表单并发布
- 进入活动详情
- 报名 / 取消报名
- 活动进行中 / 已结束
- 再次发布
- 我的活动回收

## 3. 核心原则

- 创建活动页进入时只要求已登录，不再整页拦截资料完整度
- 顶部固定只有 `个人 / 俱乐部` 两类角色卡
- 只有角色已选中且满足可发布条件后，活动表单才展开
- 联系方式统一使用手机号口径，不再采集主办方微信号
- 活动可选填封面 `coverUrl`
- 地图相关经纬度继续存储，但前端只展示地址与地点名

## 4. 创建活动链路

### 4.1 主办方视角

用户登录
-> 进入创建活动页
-> 看到个人卡和俱乐部卡
-> 选择发布角色
-> 若资料不完整，则在当前页拉起半屏补齐
-> 角色可发布后，展开正式活动表单
-> 上传活动封面（选填）
-> 配置收费、时间、报名模式、场馆与场地
-> 填写活动图文详情
-> 发布成功
-> 跳转活动详情页

### 4.2 角色分流规则

- 选择 `个人`
  - 若基础资料或手机号未完成，打开个人资料补齐半屏
  - 保存成功后自动选中 `个人` 并展开表单
- 选择 `俱乐部`
  - 若无俱乐部，打开创建俱乐部半屏
  - 若已有俱乐部但资料未完善或缺少联系人手机号，打开俱乐部完善半屏
  - 若已有多个俱乐部，通过半屏选择/确认具体俱乐部后继续

## 5. 发布主体与联系方式

### 5.1 个人发布

- 发布主体：`ownerType = PERSONAL`
- 联系方式来源：当前用户已验证手机号
- 服务端校验：必须 `contactProfileComplete = true`

### 5.2 俱乐部发布

- 发布主体：`ownerType = CLUB`
- 联系方式来源：俱乐部资料中的 `contactPhone`
- 服务端校验：
  - 当前用户必须是该俱乐部 `OWNER`
  - `contactPhone` 必须非空

## 6. 场馆与场地链路

- 场馆归属统一使用 owner 口径：
  - `ownerType = PERSONAL | CLUB`
  - `ownerId = userId | clubId`
- 个人发布时使用个人场馆
- 俱乐部发布时使用俱乐部场馆
- 场馆管理继续在创建页内以半屏承接
- 地图区只展示：
  - 地点名
  - 详细地址
  - 省 / 市 / 区

## 7. 表单关键字段

- `coverUrl`
  - 选填
  - 来自统一图片上传接口
- `descriptionRichtext`
  - 活动图文详情
  - 支持文字和图片混排
- `organizerWechat`
  - 已从活动表单与活动模型中移除

## 8. 活动状态模型

### 8.1 持久状态

`Activity.status` 只保留：

- `DRAFT`
- `PUBLISHED`
- `CANCELLED`

### 8.2 派生展示态

页面根据时间和持久状态派生：

- `报名中`
- `报名截止`
- `进行中`
- `已结束`
- `已取消`

## 9. 详情页与再次发布

- 发布成功后统一进入独立活动详情页
- 活动详情页对外展示发布主体快照和脱敏手机号
- 再次发布仍回到创建活动页
- 再次发布应继续沿用“先恢复角色，再展开表单”的口径

## 10. 对页面 PRD 的约束

- `activity-create` 必须采用“先选角色，再展开表单”
- `user-registration` 必须支持作为创建活动链路中的半屏容器
- `club-register` 必须支持作为创建活动链路中的半屏容器
- `venue-court-management` 必须遵循 owner 归属与“地址可见、经纬度隐藏”的展示口径
