# 活动全链路说明

## 1. 目的

本文用于统一 MVP 阶段活动从创建到结束的完整链路口径，作为创建活动、活动详情、我的活动、活动列表以及后续接口和类型对齐的总依据。

状态、动作、守卫条件和结果页规则，以 [活动状态与动作矩阵](./activity-state-machine.md) 为准。

当前版本只覆盖 MVP 主链路闭环，不包含结算、复盘、评价、聊天、活动结束后手工收尾等能力。

## 2. 适用范围

活动全链路覆盖以下节点：

- 创建活动
- 发布成功
- 进入活动详情
- 报名 / 取消报名
- 活动进行中
- 活动已结束
- 取消活动
- 再次发布
- 我的活动回收

## 3. 核心原则

- 活动需要独立 `活动详情页`，不由活动列表卡片临时承担详情职责
- 活动结束按 `activityEndAt` 自动判定，不提供“手动结束活动”
- 主办方最小管理动作仅包含：取消活动、调场、调容量、编辑图文详情、再次发布
- 报名和取消报名始终以个人身份处理，不存在“以俱乐部身份报名”
- 活动发布后的后续状态和操作，以活动详情页为主承接页面

## 4. 两层状态模型

### 4.1 持久状态

`Activity.status` 只保留以下枚举：

- `DRAFT`
- `PUBLISHED`
- `CANCELLED`

说明：

- `DRAFT` 表示未正式发布
- `PUBLISHED` 表示活动已发布且尚未被整体取消
- `CANCELLED` 表示活动被主办方整体取消

### 4.2 派生生命周期

页面展示层从持久状态和时间字段中派生以下生命周期：

- `草稿`
- `报名中`
- `报名截止`
- `进行中`
- `已结束`
- `已取消`

派生规则：

- `status = CANCELLED` 时，生命周期始终显示 `已取消`
- `status = DRAFT` 时，生命周期显示 `草稿`
- `status = PUBLISHED` 且当前时间小于 `cancelDeadlineAt` 时，报名状态显示 `报名中`
- `status = PUBLISHED` 且当前时间大于等于 `cancelDeadlineAt`、小于 `activityStartAt` 时，报名状态显示 `报名截止`
- `status = PUBLISHED` 且当前时间大于等于 `activityStartAt`、小于 `activityEndAt` 时，生命周期显示 `进行中`
- `status = PUBLISHED` 且当前时间大于等于 `activityEndAt` 时，生命周期显示 `已结束`

说明：

- `进行中` 和 `已结束` 是展示态，不新增为持久状态
- `报名状态` 与 `生命周期` 可并存展示，但都不反向写回 `Activity.status`

## 5. 页面职责

### 5.1 创建活动页

- 承接新建活动和再次发布
- 负责填写活动信息与图文详情并提交发布
- 在当前流程内完成报名模式、场馆选择和本次活动场地列表配置
- 若没有合适场馆，可在创建流程内直接新增或编辑场馆定位
- 发布成功后跳转到独立 `活动详情页`
- 不继续承载报名、取消报名、进行中、已结束、取消活动等后续规则

### 5.2 活动列表

- 承接活动浏览和列表筛选
- 展示活动摘要信息、当前生命周期和报名状态摘要
- 点击活动卡片进入 `活动详情页`
- 不替代独立详情页

### 5.3 活动详情页

- 是活动发布后的主承接页面
- 展示活动基础信息、图文详情、发布主体、场馆/场地、报名模式、报名名单、当前用户报名状态
- 承接关键动作结果页：报名、取消报名、调场、调容量、编辑图文详情、取消活动、再次发布

### 5.4 我的活动

- 负责回收“我发布的”活动和“我报名的”活动
- 用于按生命周期回看活动，不替代详情页
- 点击条目进入 `活动详情页`

## 6. 主链路总览

### 6.1 主办方视角

用户完成资料补全
-> 进入创建活动页
-> 确认发布身份
-> 在报名与场馆区确认报名模式
-> 选择或补充场馆定位
-> 根据报名模式配置总人数或本次活动场地列表
-> 填写活动信息与图文详情并发布
-> 跳转活动详情页
-> 在活动开始前查看报名情况
-> 视需要执行最小管理动作（调场 / 调容量 / 编辑图文详情 / 取消活动 / 再次发布）
-> 活动开始后自动进入进行中
-> 活动结束时间到后自动进入已结束
-> 活动回收到“我的活动-我发布的”

### 6.2 报名者视角

用户完成资料补全
-> 在活动列表或其他入口查看活动
-> 进入活动详情页
-> 根据报名模式报名
-> 在详情页查看自己的报名状态
-> 若在取消截止前，可取消报名
-> 活动开始后自动进入进行中
-> 活动结束后自动进入已结束
-> 活动回收到“我的活动-我报名的”

## 7. 关键分叉

### 7.1 发布主体分叉

- `PERSONAL`：活动归属个人
- `CLUB`：活动归属俱乐部
- 两种发布主体都统一进入活动详情页，报名者仍以个人身份参与

### 7.2 报名模式分叉

- `GENERAL`：统一分配，用户先报名，不预选场地，系统内不落具体场地
- `USER_SELECT_COURT`：自主选场，主办方在创建页维护本次活动场地列表，报名时需要落到具体活动场地
- 两种模式都统一支持候补、取消报名、再次发布

### 7.3 用户状态分叉

- 未报名：若仍处于可报名窗口，可直接报名
- 已报名：详情页展示当前用户报名状态
- 可取消：当前时间未超过 `cancelDeadlineAt` 且活动未开始
- 不可取消：报名截止后、活动进行中、已结束或已取消

### 7.4 管理权限分叉

- 可管理：个人活动创建者，或俱乐部活动的 `OWNER / ADMIN`
- 不可管理：其他普通报名用户、俱乐部 `MEMBER`

## 8. 最小管理动作

### 8.1 取消活动

- 仅主办方可执行
- 条件：活动未结束前可取消
- 结果：
  - `Activity.status` 变为 `CANCELLED`
  - 活动生命周期显示 `已取消`
  - 用户不再可报名
  - 已报名用户在详情页看到活动取消结果

### 8.2 调场

- 仅适用于 `USER_SELECT_COURT`
- 仅主办方可执行
- 用于在活动未结束前调整报名用户所在场地
- 调整后报名用户仍保持个人报名身份，报名记录保留

说明：

- `GENERAL / 统一分配` 不提供系统内后续分配到具体场地的动作
- 若后续要支持系统内统一编排，应作为新能力单独定义

### 8.3 调容量

- 仅主办方可执行
- 用于在活动未结束前调整统一报名总人数或单场地人数上限
- 调整后仍沿用当前候补晋升规则

### 8.4 再次发布

- 主办方可从活动详情页或我的活动进入
- 带回原活动配置，重新进入创建活动页
- 若原活动引用场地已被删除，按创建活动页既有再次发布规则处理

### 8.5 编辑图文详情

- 仅主办方可执行
- 入口位于活动详情页主办方管理区
- 条件：活动未结束且未取消
- 结果：
  - 仅更新 `descriptionRichtext`
  - 活动状态、报名状态、场地配置、容量配置不受影响
  - 更新后立即在活动详情页对所有用户生效

## 9. 我的活动回收规则

- `我发布的活动`：展示当前用户创建或可管理的活动
- `我报名的活动`：展示当前用户已报名或曾报名的活动
- 两个集合都按统一生命周期展示：报名中 / 报名截止 / 进行中 / 已结束 / 已取消
- 我的活动只做列表回收和筛选入口，具体操作和完整信息统一进入活动详情页

## 10. 与接口和类型的对齐规则

- `Activity.status` 保持 `DRAFT | PUBLISHED | CANCELLED`
- 页面展示态单独使用派生字段，例如：
  - `lifecycleStatusLabel`
  - `signupStatusLabel`
  - `isSignupOpen`
  - `isInProgress`
  - `isFinished`
- 活动详情接口至少返回：
  - 活动基础信息
  - 图文详情内容
  - 发布主体信息
  - 场馆 / 场地与报名模式
  - 报名名单
  - 当前用户报名状态
  - 当前用户是否可报名、是否可取消报名、是否可管理
  - 主办方是否可取消活动、调场、调容量、编辑图文详情、再次发布
- 我的活动接口至少支持两类集合：
  - 我发布的活动
  - 我报名的活动

## 11. 对页面 PRD 的约束

- `activity-create` 只定义创建与再次发布，不单独定义活动结束态和详情页行为
- 后续 `活动详情` 和 `我的活动` PRD 必须引用本文，不单独发明状态枚举和页面跳转口径
- `global/user-flow` 中的活动链路应与本文保持一致
