# 全局角色边界说明

## 1. 目的

本文用于统一 MVP 阶段登录主体、发布身份、俱乐部成员关系和创建活动入口的角色分流口径。

## 2. 统一原则

- 登录主体始终是用户账号，基础资料以 `UserProfile` 为准
- 创建活动入口顶部固定只有两张角色卡：`个人`、`俱乐部`
- 创建活动页进入时只校验登录，不因资料不完整整页跳转
- 报名、候补、取消报名始终以个人身份处理，不存在“以俱乐部身份报名”
- 活动对外联系方式统一为手机号口径，不再采集主办方微信号

## 3. 三层模型

### 3.1 登录主体

- 对象：`UserProfile`
- 含义：实际登录和操作的人始终是用户本人
- 作用：承接个人资料、手机号验证、俱乐部成员关系、活动创建人等数据

### 3.2 发布身份

- 枚举：`PERSONAL | CLUB`
- 含义：用户在创建活动时选择的业务发布主体
- 说明：
  - `PERSONAL` 表示以个人身份发布
  - `CLUB` 表示以某个俱乐部身份发布
  - 发布身份是业务上下文，不是新的登录角色

### 3.3 俱乐部成员关系

- 枚举：`OWNER | ADMIN | MEMBER`
- 含义：用户在某个俱乐部中的成员权限
- 当前口径：
  - `OWNER`：本轮可进入创建活动发布链路
  - `ADMIN`：数据结构保留，当前不在创建活动发布链路中开放
  - `MEMBER`：仅成员身份，不可发布活动

## 4. 创建活动两张角色卡

### 4.1 个人卡

- 显示内容：昵称、状态、手机号补齐状态
- 可发布前提：
  - 基础资料完整
  - 已完成手机号验证
- 若不满足，则点击后打开“个人资料补齐”半屏，而不是跳转整页资料页

### 4.2 俱乐部卡

- 显示内容：当前选中俱乐部、状态、说明文案
- 可发布前提：
  - 至少存在一个当前用户为 `OWNER` 的俱乐部
  - 俱乐部资料完整
  - 俱乐部联系人手机号已填写
- 若不满足，则点击后打开“创建/完善俱乐部”半屏

## 5. 最小权限矩阵

| 业务角色 | 可见个人卡 | 可见俱乐部卡 | 可个人发布 | 可俱乐部发布 | 可管理个人场馆 | 可管理俱乐部场馆 | 可报名/取消报名 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 普通用户 | 是 | 是 | 满足资料后可 | 否 | 是 | 否 | 是 |
| 俱乐部 OWNER | 是 | 是 | 满足资料后可 | 是 | 是 | 是 | 是 |
| 俱乐部 ADMIN | 是 | 是 | 满足资料后可 | 本轮否 | 是，后续可扩 | 否 | 是 |
| 俱乐部 MEMBER | 是 | 是 | 满足资料后可 | 否 | 是 | 否 | 是 |

说明：

- 页面上始终显示两张角色卡，但俱乐部卡点击后的实际承接由上下文状态决定
- 本轮 `GET /api/publish-identities` 只返回 `PERSONAL` 和 `OWNER` 俱乐部身份

## 6. 与接口和数据的对齐规则

### 6.1 创建页上下文

- `GET /api/activity-create/context` 返回的是“角色卡上下文”
- 其职责是告诉前端当前该展示“可发布 / 待补齐 / 待建档 / 待建俱乐部”中的哪种状态

### 6.2 真实可发布身份

- `GET /api/publish-identities` 返回的是“当前真正可发布的主体”
- 本轮只返回：
  - `PERSONAL`
  - 当前用户为 `OWNER` 的俱乐部

### 6.3 活动归属

- `Activity.ownerType` 与发布身份保持同一口径：`PERSONAL | CLUB`
- `Activity.ownerId`：
  - `PERSONAL` 时指向 `userId`
  - `CLUB` 时指向 `clubId`

### 6.4 场馆归属

- `Venue.ownerType = PERSONAL | CLUB`
- `Venue.ownerId = userId | clubId`
- 不再只按俱乐部维度查询场馆

## 7. 联系方式口径

- 创建活动时不再单独填写“发起人微信号”
- 个人发布：展示当前用户已验证手机号
- 俱乐部发布：展示俱乐部联系人手机号
- 活动详情默认只对外展示脱敏手机号

## 8. 与页面职责的关系

- `activity-create` 负责角色卡选择、半屏补齐和活动正式表单
- `user-registration` 负责个人资料和手机号补齐能力，既可独立页面，也可作为创建活动半屏容器
- `club-register` 负责俱乐部主体创建/完善，既可独立页面，也可作为创建活动半屏容器
