# 全局角色边界说明

## 1. 目的

本文用于统一 MVP 阶段的角色、发布身份和俱乐部成员权限口径，作为页面 PRD、接口契约、数据字典和前端类型的共同依据。

当前版本只覆盖 MVP 主链路，不扩展场馆管理员、教练、审核员等新增角色。

## 2. 统一原则

- 登录主体始终是用户账号，基础资料以 `UserProfile` 为准
- 发布活动时，用户可在可用发布身份中切换为 `PERSONAL` 或 `CLUB`
- 报名、候补、取消报名、参与活动时，始终以个人身份参与，不存在“以俱乐部身份报名”
- 俱乐部是独立业务主体，但不替代用户账号本身
- MVP 默认 `OWNER` 与 `ADMIN` 在发布活动、管理活动、管理场馆/场地上权限一致

## 3. 三层模型

### 3.1 登录主体

- 对象：`UserProfile`
- 含义：当前登录和实际操作的始终是用户本人
- 作用：承接建档资料、报名资料、俱乐部成员关系、活动创建人等用户维度数据

### 3.2 发布身份

- 枚举：`PERSONAL | CLUB`
- 含义：用户在创建活动时选择的发布主体
- 说明：
  - `PERSONAL` 表示用户以个人身份发布活动
  - `CLUB` 表示用户以某个有权限的俱乐部身份发布活动
  - 发布身份是创建活动时的业务选择，不是新的登录角色

### 3.3 俱乐部成员权限

- 枚举：`OWNER | ADMIN | MEMBER`
- 含义：用户在某个俱乐部中的成员关系权限
- 说明：
  - `OWNER / ADMIN` 可代表俱乐部发布活动，并管理该俱乐部活动、场馆和场地
  - `MEMBER` 仅表示俱乐部成员身份，可参与活动，但不能代表俱乐部发布活动

## 4. 四类业务角色

### 4.1 普通用户

- 定义：完成基础建档的用户
- 能力：
  - 浏览活动
  - 报名 / 取消报名
  - 查看我的活动
  - 以 `PERSONAL` 身份发布活动

### 4.2 个人发布者

- 定义：普通用户在创建活动时选择 `PERSONAL` 作为发布主体时的业务状态
- 边界：
  - 不是独立账号角色
  - 不影响用户报名时仍以个人身份参与
  - 可与俱乐部管理员身份同时存在于同一用户身上

### 4.3 俱乐部管理员

- 定义：用户在某俱乐部中的成员权限为 `OWNER` 或 `ADMIN`
- 能力：
  - 在创建活动时看到对应俱乐部作为可选发布身份
  - 以 `CLUB` 身份发布活动
  - 管理该俱乐部下活动
  - 管理该俱乐部下场馆与场地

### 4.4 俱乐部成员

- 定义：用户在某俱乐部中的成员权限为 `MEMBER`
- 能力与限制：
  - 可作为普通用户报名和参与活动
  - 不可代表俱乐部发布活动
  - 不出现在当前用户可用发布身份列表中

## 5. 最小权限矩阵

| 业务角色 | 可看到个人发布身份 | 可看到俱乐部发布身份 | 可个人发布活动 | 可俱乐部发布活动 | 可管理俱乐部场馆/场地 | 可报名/取消报名 | 可管理已发布活动 |
| -------- | ------------------ | -------------------- | -------------- | ---------------- | --------------------- | --------------- | ---------------- |
| 普通用户 | 是 | 否 | 是 | 否 | 否 | 是 | 仅自己创建或自己可管理的活动 |
| 个人发布者 | 是 | 视成员关系而定 | 是 | 视成员关系而定 | 视成员关系而定 | 是 | 仅当前发布身份下自己可管理的活动 |
| 俱乐部管理员 | 是 | 是 | 是 | 是 | 是 | 是 | 是，限所属俱乐部活动 |
| 俱乐部成员 | 是 | 否 | 是 | 否 | 否 | 是 | 否 |

说明：

- 所有用户默认都应有一个 `PERSONAL` 发布身份
- `CLUB` 发布身份仅对当前用户为该俱乐部 `OWNER / ADMIN` 时可见
- 报名、候补、取消报名始终作用于 `userId`，不作用于 `clubId`

## 6. 与类型和接口的对齐规则

### 6.1 发布身份

- `PublishIdentity.identityType` 统一使用 `PERSONAL | CLUB`
- 页面文案继续展示为“个人 / 俱乐部”
- 若接口需要展示当前用户可用身份，返回的是“可发布主体”，不是“所有成员关系”

### 6.2 活动归属

- `Activity.ownerType` 与发布身份使用同一口径：`PERSONAL | CLUB`
- `Activity.ownerId` 在 `PERSONAL` 时指向 `userId`，在 `CLUB` 时指向 `clubId`
- 活动详情中的组织者可展示为个人或俱乐部，但报名名单中的参与者始终展示为个人

### 6.3 俱乐部成员关系

- `ClubMember.role` 继续使用 `OWNER | ADMIN | MEMBER`
- `MEMBER` 是成员关系，不是发布身份，不应直接映射为创建活动页的可选项

## 7. MVP 场景约束

- 未加入任何俱乐部的用户，只看到 `PERSONAL` 发布身份
- `OWNER / ADMIN` 用户可在 `PERSONAL` 与所属俱乐部之间切换发布主体
- `MEMBER` 用户可以参加俱乐部活动，但不能代表俱乐部发布活动
- 用户可以先以俱乐部身份发布活动，再以个人身份去报名其他活动，两者不冲突

## 8. 与页面职责的关系

- `mvp-overview` 只描述目标用户和产品范围，不重复展开发布身份或成员权限规则
- `club-register` 负责俱乐部主体创建、成员关系来源及创建成功后的身份承接
- `activity-create` 负责消费可用发布身份并完成活动创建，不单独重新定义角色体系
