# 场馆与场地管理产品文档

面向对象：UI 设计、前端开发、后端开发

版本：v1.5

更新时间：2026-04-22

## 页面目标

场馆与场地管理页用于低频维护活动发布依赖的场馆资料和可复用场地资料，作为创建活动页之外的补充维护入口。

当前版本目标：

- 支持新增和编辑场馆
- 支持地图选点回填场馆定位
- 支持新增和编辑场地
- 支持删除场地
- 支持查看并整理某场馆下可复用的场地资料

## 在主链路中的位置

该页面位于主链路中的补充维护分叉：

`登录 -> 进入创建活动页 -> 选择个人或俱乐部角色 -> 必要时进入场馆与场地管理页 -> 返回创建活动表单 -> 活动详情 -> 报名/取消报名 -> 我的活动回收`

它服务于创建活动中的低频资料维护，不属于活动详情管理动作，也不是创建前必经步骤。

## 前置条件

- 当前用户已登录
- 当前用户已选定创建活动发布角色，或从“我的”独立入口进入
- 当前俱乐部场馆链路仅对俱乐部 `OWNER` 开放
- 场馆、场地、活动快照的资源边界统一引用全局文档

## 页面职责

- 维护场馆主数据
- 低频维护可复用场地资料
- 为创建活动页提供可选场馆和可复用场地来源
- 保证场地删除不影响历史活动快照

## 非职责边界

- 不负责活动详情中的调场
- 不负责活动详情中的调容量
- 不负责活动详情中的取消活动
- 不负责活动发布后的报名名单展示
- 不负责历史活动快照的反向修改

## 入口与回跳

建议入口：

- `activity-create` 页面中的 `管理场馆与场地`
- 后续 `profile` / “我的”中的场馆管理入口

回跳规则：

- 若从 `activity-create` 进入，保存成功后返回创建活动页
- 返回后刷新场馆列表与可选场地
- 若本次新增了新场馆，创建活动页默认切换到新建场馆
- 若编辑的是当前场馆，尽量保留当前表单中的已配场地列表

## 字段与状态

### 场馆字段

- `name`
  - 必填
- `shortName`
  - 选填，不填写时默认同场馆名称
- `navigationName`
  - 选填，不填写时默认同场馆名称
- `province / city / district / address / latitude / longitude`
  - 地址和内部定位坐标由地图选点回填
  - `province / city / district` 可由选点结果补充，或允许手动修正

### 场地字段

- `courtCode`
  - 必填
  - 同一场馆下唯一
- `courtName`
  - 必填
- `defaultCapacity`
  - 必填
  - 必须大于 0
- `defaultFeeCents`
  - 必填
  - 前端按元输入，提交时转分
- `status`
  - `ACTIVE / DISABLED`
  - 保留为实现兼容字段，不作为 MVP 主要交互概念

### 状态说明

- 页面不重新定义活动状态
- 已删除场地不再用于新活动，但不影响历史活动快照
- 若实现层需要兼容历史引用，可将“删除场地”映射为停用或归档

## 关键交互

- 点击地图定位默认拉起微信小程序原生 `wx.chooseLocation`
- 默认回填地点名、详细地址和内部经纬度
- `province / city / district` 若未完整返回，可由用户手动补充或修正
- 对用户只展示地点名、详细地址和省市区，不展示经纬度文案或数值
- 新建未保存场地可直接删除
- 已保存场地在产品上统一表达为“删除”；实现层可映射为 `DISABLED` 或软删除
- 点击 `保存场馆` 时需要同时提交：
  - 场馆基础信息
  - 定位信息
  - 场地列表
- 保存成功后返回来源页，并刷新创建活动页中的可选数据

新增场馆步骤：

1. 用户从创建活动页或“我的”入口进入场馆与场地管理页。
2. 页面根据来源确定当前 `ownerType / ownerId`。
3. 用户填写场馆名称、简称和导航名。
4. 用户点击地图定位并选择地点。
5. 页面回填地点名、详细地址、省市区和内部经纬度。
6. 用户按需新增场地。
7. 用户点击保存场馆。
8. 系统创建场馆并保存场地列表。
9. 若来源为创建活动页，返回后默认选中新建场馆。

编辑场馆步骤：

1. 用户进入已有场馆的编辑状态。
2. 页面回填场馆基础信息、定位信息和场地列表。
3. 用户调整场馆名称、地址或定位。
4. 用户保存场馆。
5. 系统更新场馆主数据。
6. 若编辑的是创建活动页当前选中场馆，返回后刷新展示但尽量保留活动表单内容。

增删场地步骤：

1. 用户在场地列表区点击新增场地。
2. 页面新增一条未保存场地表单。
3. 用户填写场地号、场地名称、默认人数和默认费用。
4. 用户可删除未保存场地。
5. 用户可对已保存场地点击删除。
6. 产品上统一表达为删除；实现层可映射为停用或软删除。
7. 用户保存场馆后，场地列表变更生效。
8. 已删除场地不再进入新活动可选列表，但历史活动快照不受影响。

保存并返回创建页步骤：

1. 用户点击保存场馆。
2. 页面校验场馆必填项、定位信息和场地字段。
3. 页面提交场馆基础信息、定位信息和场地列表。
4. 保存成功后返回来源页。
5. 创建活动页刷新当前发布主体下的可选场馆和场地。
6. 若本次新增了新场馆，创建活动页默认选中新场馆。
7. 用户继续配置本次活动报名模式和场地列表。

## 异常与空状态

## 验收标准

- 用户可成功新增一个带定位信息的场馆
- 用户可单独维护场馆定位与基础资料，不强制先配置完整场地
- 用户可在一个场馆下新增、编辑、删除多片场地
- 已删除场地不再出现在新建活动可选列表中，但历史活动快照仍可正常展示
- 历史活动仍能显示发布时的场地快照
- 创建活动时仍可带出当前场馆下可复用的场地与默认人数
- 场馆定位默认基于微信小程序原生选点能力，不依赖腾讯地图插件
- 地图信息对用户只显示地点名和地址，不显示经纬度
- 本页作为补充维护入口存在，不再是创建前默认前置步骤
