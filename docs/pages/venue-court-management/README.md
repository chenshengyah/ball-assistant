# 场馆与场地管理产品文档

面向对象：UI 设计、前端开发、后端开发

版本：v1.1

更新时间：2026-04-08

## 1. 页面目标

场馆与场地管理用于维护活动发布依赖的场馆主数据和场地主数据。

当前版本目标：

- 支持新增场馆
- 支持编辑场馆基础信息
- 支持通过地图选择场馆定位
- 支持维护场地下的多片场地
- 支持为每片场地维护默认人数和默认费用
- 支持停用历史场地，不影响历史活动快照

当前版本不包含：

- 场馆列表高级筛选
- 场馆图片上传
- 多城市切换
- 场地拖拽排序
- 场地批量导入
- 场地物理删除

## 2. 页面定位

该页面是一个“主数据维护页”，服务于主办方或管理员维护活动发布所依赖的基础数据。

设计原则：

- 场馆是主数据
- 场地是从属于场馆的主数据
- 活动发布时引用场地，但要保留活动自己的场地快照
- 活动里的场地描述属于活动发布配置，不属于场馆或场地主数据
- 场地停用后，不再出现在新建活动中
- 历史活动仍展示发布时的场地快照，不受主数据变更影响
- 移动端优先，交互以卡片式表单为主，不采用复杂表格

## 3. 关系说明

数据关系如下：

- 一个场馆包含多片场地
- 一场活动只属于一个场馆
- 活动发布时，会从场馆场地生成活动场地快照

推荐模型：

- `Venue 1 -> N VenueCourt`
- `Venue 1 -> N Activity`
- `Activity 1 -> N ActivityCourt`
- `ActivityCourt N -> 1 VenueCourt`

说明：

- `VenueCourt` 保存主数据
- `ActivityCourt` 保存活动发布时的快照数据
- `ActivityCourt` 中需要保留 `courtCodeSnapshot`、`courtNameSnapshot`、`capacity`、`feeCents` 等字段
- 如活动需要补充“20:00 开灯”“靠里侧入口”等临时说明，应写入 `ActivityCourt` 快照而不是 `VenueCourt`

## 4. 页面结构

页面分为 3 个主模块和 1 个底部操作区。

### 4.1 页面标题区

- 标题：`场馆与场地管理`
- 副标题：`维护活动发布所需的场馆、定位和场地配置`

### 4.2 模块一：场馆基础信息

包含字段：

- 场馆名称
- 简称
- 导航名
- 地图定位入口
- 区域
- 详细地址

### 4.3 模块二：场地列表

包含内容：

- 场地卡片列表
- 每片场地的名称、人数、费用
- 删除或停用入口
- `新增场地` 按钮

### 4.4 底部操作区

- 取消
- 保存场馆

## 5. 场馆字段定义

### 5.1 场馆名称

- 类型：单行输入
- 是否必填：是
- 占位文案：`例如：浦东金桥羽毛球馆`
- 说明：作为场馆主名称展示

### 5.2 简称

- 类型：单行输入
- 是否必填：否
- 占位文案：`例如：金桥馆`
- 规则：不填写时，默认等于场馆名称

### 5.3 导航名

- 类型：单行输入
- 是否必填：否
- 占位文案：`默认同场馆名称`
- 规则：不填写时，默认等于场馆名称
- 说明：用于地图导航、打开地图时展示

### 5.4 地图定位

- 类型：按钮触发地图选点
- 是否必填：建议必填
- 按钮文案：`选择地图位置`
- 成功后回填字段：
  - `latitude`
  - `longitude`
  - `address`
  - `province`
  - `city`
  - `district`
  - `navigationName`

说明：

- 使用腾讯地图 API 或微信位置能力接入
- 前端负责把地图返回结果拆分为结构化字段
- 回填后允许用户手动修改导航名、区域和详细地址
- 若地图返回的行政区拆分不准确，以用户手动修改为准

### 5.5 区域

- 类型：单行输入
- 是否必填：否
- 占位文案：`例如：浦东新区`
- 规则：地图返回后自动回填，可编辑

### 5.6 详细地址

- 类型：单行输入或多行输入
- 是否必填：是
- 占位文案：`例如：金桥路 588 号 B1`
- 规则：地图返回后自动回填，可编辑

## 6. 场地字段定义

场地以“卡片列表”形式维护，每片场地为一张卡片。

### 6.1 场地序号

- 存储字段：`courtCode`
- 类型：系统自动生成，只读展示
- 展示形式：`1`、`2`、`3`
- 是否允许手动编辑：否

说明：

- `courtCode` 保留在数据模型中，作为稳定编码使用
- UI 不暴露输入框，避免移动端输入负担
- 删除场地后，前端展示序号可随当前顺序重新展示
- 后端真实 `id` 不依赖展示序号

### 6.2 场地名称

- 存储字段：`courtName`
- 类型：单行输入
- 是否必填：是
- 占位文案：`例如：1号场 / 靠窗场 / 训练场`

### 6.3 默认人数

- 存储字段：`defaultCapacity`
- 类型：数字输入
- 是否必填：是
- 单位：人
- 默认值：`4`
- 规则：必须为大于 `0` 的整数

说明：

- 该字段是场馆主数据里的默认值
- 创建活动时可自动带出，并允许按活动单独调整

### 6.4 默认费用

- 存储字段：`defaultFeeCents`
- 类型：数字输入
- 是否必填：是
- UI 单位：元
- 存储单位：分
- 默认值：`0`
- 规则：必须为大于等于 `0` 的合法数字

说明：

- 该字段是场馆主数据里的默认值
- 当前版本不接入创建活动页主流程
- 作为后续结算、运营参考或活动发布默认值预留
- 前端展示按元输入，提交时转换为分

### 6.5 场地状态

- 存储字段：`status`
- 枚举值：`ACTIVE | INACTIVE`
- UI 表达：
  - 启用中
  - 已停用

说明：

- 已停用场地不会出现在新建活动中
- 已停用场地不影响历史活动的场地快照展示

## 7. 移动端交互方案

### 7.1 页面整体交互

- 页面采用纵向滚动表单
- 场馆基础信息位于上半部分
- 场地列表位于下半部分
- 底部提供固定操作按钮

### 7.2 场地卡片样式

每张场地卡片包含：

- 卡片标题：`场地 1`、`场地 2`
- 序号展示：只读
- 场地名称输入框
- 人数输入框
- 费用输入框
- 删除或停用操作

建议交互：

- 新建中的卡片可直接删除
- 已保存卡片优先使用“停用”
- 停用后的卡片保留展示，但置灰并显示状态标签

### 7.3 新增场地

- 在场地列表底部放置 `新增场地` 按钮
- 点击一次，新增一张场地卡片
- 新卡片默认值：
  - `courtCode = 当前顺序`
  - `courtName = ""`
  - `defaultCapacity = 4`
  - `defaultFeeCents = 0`
  - `status = ACTIVE`

### 7.4 删除与停用

规则分两类：

- 新建未保存的场地：允许直接删除卡片
- 已保存的场地：不建议物理删除，改为停用

原因：

- 已保存场地可能被历史活动引用
- 停用更符合“主数据维护”语义
- 可避免历史活动数据断链

### 7.5 场地排序

- 当前版本默认按新增顺序排列
- 暂不支持拖拽排序
- 若删除中间卡片，UI 可按当前顺序重绘展示编号

## 8. 保存规则

### 8.1 保存场馆

保存时需要提交：

- 场馆基础信息
- 场馆定位信息
- 场地列表

建议后端接收结构：

```ts
type VenueSaveInput = {
  name: string;
  shortName: string;
  navigationName: string;
  province: string;
  city: string;
  district: string;
  address: string;
  latitude: number;
  longitude: number;
  courts: Array<{
    id?: string;
    courtCode: string;
    courtName: string;
    defaultCapacity: number;
    defaultFeeCents: number;
    sortOrder: number;
    status: "ACTIVE" | "INACTIVE";
  }>;
};
```

### 8.2 费用转换规则

- UI 输入：元
- 提交接口：分
- 示例：
  - UI 输入 `35`
  - 提交 `3500`

### 8.3 活动创建时带出规则

创建活动时，若用户选择某个场馆，则默认带出该场馆下启用中的场地：

- `courtCode -> courtCodeSnapshot`
- `courtName -> courtNameSnapshot`
- `defaultCapacity -> capacity`

说明：

- 活动创建页允许主办方再做一次活动级微调
- 后续主数据变化不应反向覆盖已发布活动
- 当前版本只带出场地默认人数，不带出场地默认费用

## 9. 校验规则

### 9.1 场馆基础信息

- 场馆名称不能为空
- 详细地址不能为空
- 建议必须选择地图定位

### 9.2 场地列表

- 至少需要 1 片启用中的场地
- 场地名称不能为空
- 默认人数必须大于 `0`
- 默认费用不能小于 `0`
- 同一场馆下 `courtCode` 不能重复

## 10. 错误提示文案

- 场馆名称为空：`请填写场馆名称`
- 未填写详细地址：`请填写场馆地址`
- 未选择定位：`请先选择场馆定位`
- 没有启用中的场地：`请至少新增一片场地`
- 场地名称为空：`请填写场地名称`
- 人数不合法：`人数需大于 0`
- 费用不合法：`费用不能小于 0`
- 编码重复：`场地序号重复，请刷新后重试`

## 11. 设计与开发备注

- UI 不展示 `courtCode` 输入框，只把它作为序号展示
- 前端编辑态可以使用数组对象维护场地列表
- 后端仍建议为每片场地生成稳定 `id`
- 费用统一按“前端元、后端分”处理
- 场地默认费用当前不进入创建活动页，只保留在场馆主数据层
- 若后续接入独立场馆后台页，本 PRD 可直接沿用

## 12. 验收标准

- 用户可成功新增一个带定位信息的场馆
- 用户可在一个场馆下新增多片场地
- 用户可为每片场地填写名称、人数、费用
- 用户可删除新建未保存的场地
- 用户可停用已保存场地
- 停用场地后，不再出现在新建活动可选列表中
- 历史活动仍能显示原有场地快照
- 创建活动时，能自动带出场地默认人数
