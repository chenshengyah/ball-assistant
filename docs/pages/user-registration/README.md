# 完善信息 / 资料补全产品文档

面向对象：UI 设计、前端开发、后端开发

版本：v1.4

更新时间：2026-04-22

说明：

- 本文档的产品名称统一使用 `完善信息 / 资料补全`
- 现有文件路径 `user-registration` 仅作为工程命名保留

## 页面目标

完善信息页用于在用户触发关键动作前补齐最小资料闭环，确保后续发布活动、报名活动、注册俱乐部、进入个人相关入口时，都基于统一的头像、昵称、性别与联系方式口径运行。

本轮补充说明：

- 该能力既可以独立页面承接，也可以在创建活动链路中以半屏形式承接
- 创建活动和报名活动来源都要求完成手机号验证
- 创建活动页不再在进入前整页跳转到本页，而是在用户选择个人角色后按需以半屏拉起

它不是“首次进入小程序强制拦截页”，而是“角色相关动作触发前的资料补全页”。

当前版本目标：

- 支持首次补齐基础资料
- 支持补齐头像、昵称与性别
- 在有业务必要时补齐联系方式
- 支持按来源页回跳
- 为后续“我的”资料展示与编辑提供统一数据源

## 在主链路中的位置

该页面位于整条 MVP 主链路的动作前补全节点：

`完善信息 -> 确认发布身份 -> 俱乐部注册 -> 创建活动（必要时补充场馆与场地） -> 活动详情 -> 报名/取消报名 -> 我的活动回收`

未完成资料补全的用户仍可浏览首页、活动列表、活动详情和“我的”游客态；只有进入创建、报名、我的活动、俱乐部入口等角色相关动作时，才跳到本页。

## 前置条件

- 当前用户已通过静默登录或动作触发登录拿到最小账号主体
- 系统可判断当前用户是否缺少基础资料或联系方式
- 资料字段口径以 [数据字典](../../global/data-dictionary.md) 和 [API Contract](../../global/api-contract.md) 为准

## 页面职责

- 采集并提交 `avatarUrl`
- 采集并提交 `nickname`
- 采集并提交 `gender`
- 在创建活动或报名活动前补齐并验证手机号
- 在用户触发角色相关动作后补齐基础资料
- 作为其他业务页面的前置拦截页，完成后回到来源页继续流程

## 非职责边界

- 不负责活动报名
- 不负责俱乐部注册
- 不负责活动发布
- 不负责“我的活动”列表回收
- 不承担完整个人中心编辑页职责

## 入口与回跳

建议入口：

- 用户尝试创建活动，但缺少基础资料或联系方式
- 用户尝试报名活动，但缺少基础资料或联系方式
- 用户尝试取消报名，但缺少基础资料
- 用户尝试进入“我的活动”，但缺少基础资料
- 用户尝试注册俱乐部，但缺少基础资料
- 用户从“我的”页主动补齐资料

提交成功后的回跳规则：

- 若存在来源页，则返回来源页继续原动作
- 若无明确来源页，则进入首页或“我的”页默认入口
- 来源页至少应支持以下来源标识：
  - `activity-create`
  - `activity-detail`
  - `my-activities`
  - `club-register`
  - `profile`

## 核心模块

### 1. 页面标题区

- 标题按来源动态变化：
  - 创建活动 / 报名：`完善资料与联系方式`
  - 我的活动 / 俱乐部入口 / 主动补资料：`先补齐基础资料`
- 副标题按来源动态变化，并明确是否要求联系方式
- 页面需要让用户感知：
  - 为什么来到这里
  - 保存后会回到哪里

### 2. 基础资料区

- 头像：微信 `chooseAvatar`
- 昵称
  - 使用 `input type="nickname"`
- 性别

### 3. 联系方式区

- 仅在以下来源展示并要求验证：
  - 创建活动
  - 报名活动
- 产品目标口径：
  - 使用微信 `button open-type="getPhoneNumber"`
  - 前端只上传 `code`
  - 后端换取手机号后保存，不在活动详情页默认明文公开
- 当前实现阶段：
  - 若尚未真正落手机号能力，PRD 中也要为 UI 保留该区块和已验证/未验证状态位
  - UI 设计稿按状态位出图，不要求 UI 自行猜测是否显示

### 4. 提交区

- `提交`
- 提交按钮文案按来源动态变化，例如：
  - `保存并继续发布`
  - `保存并继续报名`
  - `保存并继续查看`
  - `保存资料`

### 5. 来源态说明

- 创建活动来源：
  - 保存后继续进入创建活动
- 活动详情报名来源：
  - 保存后返回活动详情并继续刚才动作
- 我的活动来源：
  - 保存后继续打开我的活动
- 俱乐部入口来源：
  - 保存后继续进入俱乐部相关流程
- 我的页主动补资料来源：
  - 保存后返回个人入口或停留当前来源容器

## 字段与状态

### 字段

- `avatarUrl`
  - 通过微信头像能力选择
  - 非空时回填预览
- `nickname`
  - 单行输入
  - 必填
  - 去除首尾空格后不能为空
  - 建议长度 1 到 20 字
- `gender`
  - 单选 picker / radio
  - 必填
  - 枚举仅允许 `MALE / FEMALE`
- `phoneNumber`
  - 仅在创建活动、报名活动来源强制
  - 通过微信手机号快速验证组件采集
  - 不允许前端直接手填

### 状态

- 页面不单独发明领域状态
- 用户资料字段、枚举和值域统一引用 [数据字典](../../global/data-dictionary.md)
- 资料完整度拆分为：
  - `baseProfileComplete`
  - `contactProfileComplete`

## 关键交互

- 进入页面时，若已有资料，则回填头像、昵称和性别
- 页面 copy、提交按钮文案、来源说明按入口来源动态变化
- 创建活动 / 报名来源下，若手机号已验证，则显示已验证状态
- 点击 `提交` 时：
  1. 校验昵称
  2. 校验性别
  3. 若当前动作要求手机号，则必须先完成手机号验证
  4. 调用基础资料保存接口
  5. 成功后优先恢复待继续动作
  6. 若无待继续动作，再按来源回跳
- 成功提示建议：`资料已完善`

## 异常与空状态

- 昵称为空：`请输入昵称`
- 性别未选择：`请选择性别`
- 用户拒绝手机号验证：`你还没有完成手机号验证`
- 小程序主体无该能力或后端未配置：展示后端返回的明确错误
- 保存失败：`保存失败，请稍后重试`
- 若来源页已失效，提交成功后进入首页默认入口

## 验收标准

- 未补全资料用户可完成头像昵称、性别填写并保存成功
- 页面可按来源改变标题、副标题、提交按钮文案和来源说明
- 创建活动、报名来源下必须先完成联系方式补齐才能继续
- 补全成功后可返回原来源页继续流程
- 创建活动、报名活动、俱乐部注册都不再重复采集头像昵称与性别
- 任一业务页面都能把“动作前补资料再继续”的规则收口到本页
- 该页不承担完整个人中心编辑页职责
