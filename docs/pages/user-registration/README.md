# 用户建档 / 注册产品文档

面向对象：UI 设计、前端开发、后端开发

版本：v1.1

更新时间：2026-04-13

## 页面目标

用户建档 / 注册页用于完成当前用户最小资料闭环，确保后续发布活动、报名活动、注册俱乐部都基于统一的 `nickname` 与 `gender` 口径运行。

它不是“首次进入小程序强制拦截页”，而是“角色相关动作触发前的补资料页”。

当前版本目标：

- 支持首次建档
- 支持补齐昵称与性别
- 支持按来源页回跳
- 为后续“我的”资料展示与编辑提供统一数据源

## 在主链路中的位置

该页面位于整条 MVP 主链路的动作前补全节点：

`用户建档 -> 选择发布身份 -> 俱乐部注册 -> 场馆/场地准备 -> 创建活动 -> 活动详情 -> 报名/取消报名 -> 我的活动回收`

未完成建档的用户仍可浏览首页、活动列表、活动详情和“我的”游客态；只有进入创建、报名、我的活动、俱乐部入口等角色相关动作时，才跳到本页。

## 前置条件

- 当前用户已通过静默登录或动作触发登录拿到最小账号主体
- 系统可判断当前用户是否缺少 `nickname` 或 `gender`
- 资料字段口径以 [数据字典](../../global/data-dictionary.md) 和 [API Contract](../../global/api-contract.md) 为准

## 页面职责

- 采集并提交 `nickname`
- 采集并提交 `gender`
- 在用户触发角色相关动作后补齐基础资料
- 作为其他业务页面的前置拦截页，完成后回到来源页继续流程

## 非职责边界

- 不负责活动报名
- 不负责俱乐部注册
- 不负责活动发布
- 不负责“我的活动”列表回收
- 不承担完整资料编辑中心职责

## 入口与回跳

建议入口：

- 用户尝试创建活动，但缺少基础资料
- 用户尝试报名活动，但缺少基础资料
- 用户尝试取消报名，但缺少基础资料
- 用户尝试进入“我的活动”，但缺少基础资料
- 用户尝试注册俱乐部，但缺少基础资料
- 用户从“我的”页主动补齐资料

提交成功后的回跳规则：

- 若存在来源页，则返回来源页继续原动作
- 若无明确来源页，则进入首页或“我的”页默认入口
- 来源页至少应支持以下来源标识：
  - `activity-create`
  - `activity-detail`
  - `my-activities`
  - `club-register`
  - `profile`

## 核心模块

### 1. 页面标题区

- 标题：`完善个人资料`
- 副标题：`完成基础建档后，即可继续报名、发布和身份切换`

### 2. 基础资料区

- 昵称
- 性别

### 3. 底部操作区

- `提交`

## 字段与状态

### 字段

- `nickname`
  - 单行输入
  - 必填
  - 去除首尾空格后不能为空
  - 建议长度 1 到 20 字
- `gender`
  - 单选 picker / radio
  - 必填
  - 枚举仅允许 `MALE / FEMALE`

### 状态

- 页面不单独发明领域状态
- 用户资料字段、枚举和值域统一引用 [数据字典](../../global/data-dictionary.md)

## 关键交互

- 进入页面时，若已有资料，则回填已有昵称和性别
- 点击 `提交` 时：
  1. 校验昵称
  2. 校验性别
  3. 调用保存接口
  4. 成功后优先恢复待继续动作
  5. 若无待继续动作，再按来源回跳
- 成功提示建议：`资料已完善`

## 异常与空状态

- 昵称为空：`请输入昵称`
- 性别未选择：`请选择性别`
- 保存失败：`保存失败，请稍后重试`
- 若来源页已失效，提交成功后进入首页默认入口

## 验收标准

- 未建档用户可完成昵称与性别填写并保存成功
- 建档成功后可返回原来源页继续流程
- 创建活动、报名活动、俱乐部注册都不再重复采集昵称与性别
- 任一业务页面都能把“动作前补资料再继续”的规则收口到本页
- 首页、活动列表、活动详情、“我的”游客态无需先经过本页
