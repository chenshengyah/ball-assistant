## 1. 页面目标

用户建档 / 注册页用于承接用户首次进入小程序后的基础资料完善，确保后续活动报名、发布身份、俱乐部注册等流程都基于统一的用户资料口径运行。

当前版本目标：

- 支持首次建档
- 支持填写昵称
- 支持选择性别
- 支持提交成功后进入首页或来源页
- 为后续“我的”资料展示与编辑提供统一数据源

当前版本不包含：

- 手机号登录体系
- 第三方账号绑定
- 头像裁剪与上传
- 复杂资料编辑历史
- 完整的运行时拦截规则

## 2. 页面定位

该页面是一个“基础建档页”，只负责完成用户最小资料闭环，不负责活动报名、俱乐部注册或个人主页展示。

边界说明：

- 本页负责采集 `nickname`、`gender`
- `gender` 为必填字段，枚举仅允许 `MALE / FEMALE`
- 当前 `profile` 页面不承担首次建档职责，后续仅作为资料展示 / 编辑预留入口
- 俱乐部注册页不重复采集用户昵称或性别

## 3. 进入时机

用户应在以下时机进入本页：

- 首次进入小程序且当前用户尚未完成基础建档
- 系统检测到当前账号缺少 `nickname` 或 `gender`

提交成功后的返回规则：

- 若存在来源页，则返回来源页
- 若无明确来源页，则进入首页

## 4. 页面结构

页面分为 2 个主模块和 1 个底部操作区。

### 4.1 页面标题区

- 标题：`完善个人资料`
- 副标题：`完成基础建档后，即可继续报名、发布和身份切换`

### 4.2 模块一：基础资料

包含字段：

- 昵称
- 性别

### 4.3 底部操作区

- 提交

## 5. 字段定义

### 5.1 昵称

- 存储字段：`nickname`
- 类型：单行输入
- 是否必填：是
- 占位文案：`请输入昵称`
- 规则：
  - 去除首尾空格后不能为空
  - 长度建议 1 到 20 字

### 5.2 性别

- 存储字段：`gender`
- 类型：单选 picker / radio
- 是否必填：是
- 枚举值：
  - `MALE`
  - `FEMALE`

UI 文案建议：

- `男`
- `女`

规则：

- 当前版本不允许留空
- 当前版本不提供“未设置”“其他”选项
- 提交时必须落为标准枚举值 `MALE / FEMALE`

## 6. 数据结构建议

推荐当前用户基础资料至少包含以下字段：

```ts
type UserGender = "MALE" | "FEMALE";

type UserProfile = {
  userId: string;
  nickname: string;
  gender: UserGender;
  avatarUrl?: string;
  avatarColor?: string;
};
```

说明：

- `gender` 为必填
- `avatarUrl`、`avatarColor` 可按展示能力逐步补齐
- 业务对象应优先通过 `userId` 关联 `UserProfile`

## 7. 接口约定

页面至少依赖以下接口：

- `GET /api/users/me`：获取当前用户资料
- `PUT /api/users/me/profile`：提交或更新用户资料

请求与响应都必须包含：

- `nickname`
- `gender`

## 8. 提交与成功态

点击 `提交` 时：

1. 校验昵称非空
2. 校验性别已选择
3. 调用保存接口
4. 保存成功后进入来源页或首页

成功提示建议：

- `资料已完善`

## 9. 校验规则

- 昵称为空：提示 `请输入昵称`
- 性别未选择：提示 `请选择性别`
- 保存失败：提示 `保存失败，请稍后重试`
