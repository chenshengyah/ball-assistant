# 俱乐部创建与注册产品文档

面向对象：UI 设计、前端开发、后端开发

版本：v1.1

更新时间：2026-04-08

相关文档：

## 1. 页面目标

俱乐部创建与注册用于让用户创建一个可被平台识别的俱乐部主体，并完成与活动发布身份的衔接，供后续：

- 以俱乐部身份发布活动
- 展示俱乐部基础资料
- 沉淀俱乐部联系方式和默认归属地
- 在创建活动时切换为俱乐部身份继续发布

当前版本目标：

- 支持创建俱乐部基础资料
- 支持上传封面和 Logo
- 支持通过腾讯地图选择俱乐部地点
- 支持填写联系人与联系方式
- 支持提交成功后生成俱乐部主体
- 支持从创建活动页进入并在注册成功后回跳来源页
- 支持注册成功后自动选中新创建的俱乐部身份

当前版本不包含：

- 俱乐部审核后台
- 俱乐部主页装修
- 俱乐部成员邀请
- 多品类配置
- 营业时间、收费规则、认证材料

## 2. 页面定位

该文档定义的是“俱乐部主体创建链路”，以注册页为核心，同时覆盖与活动发布页之间最小必要的身份承接规则。

设计原则：

- 俱乐部是独立主体，不等于用户账号
- 用户昵称、性别等基础资料由[用户建档/注册页面](../user-registration/README.md)负责，本页不重复采集
- 注册页只负责创建俱乐部资料
- 活动发布页负责选择发布主体并创建活动
- 俱乐部地点是默认归属地，不等于具体活动场馆
- 表单尽量一次填清，避免过度拆步骤
- 注册成功后应尽量不中断用户原有发布流程

## 3. 适用场景

用户可能从以下场景进入：

- 在创建活动时，希望切换为俱乐部身份，但当前还没有俱乐部
- 在“我的”页面主动注册俱乐部
- 后续在“我的俱乐部”中补全资料

页面本身不依赖具体入口，但需要支持“提交成功后按来源回跳”。

### 3.1 与活动发布的身份关系

- 登录主体始终是用户账号
- 发布主体可以是“个人”或“俱乐部”
- 创建活动时，“发布身份”字段保留单选
- 默认一定有一个“个人”身份
- 若当前用户是某俱乐部的 `OWNER / ADMIN`，可额外看到对应俱乐部
- 若当前用户没有可用俱乐部身份，应在创建活动页提供 `注册俱乐部` 入口

### 3.2 从创建活动页进入时的承接规则

- 用户从创建活动页进入注册页时，注册成功后应返回创建活动页
- 返回时至少带回：
  - 新创建的 `clubId`
  - 来源标识
- 若来源为创建活动页，上层页面需自动选中新注册俱乐部作为发布身份
- 返回创建活动页后，用户先前已填写的活动表单内容应保留

## 4. 页面结构

页面分为 4 个主模块和 1 个底部操作区。

### 4.1 页面标题区

- 标题：`注册俱乐部`
- 副标题：`完善俱乐部资料后，即可作为俱乐部主体使用`

### 4.2 模块一：基础资料

包含字段：

- 俱乐部封面
- 俱乐部 Logo
- 俱乐部名称
- 俱乐部品类
- 俱乐部简介

### 4.3 模块二：位置信息

包含字段：

- 地图选点入口
- 地点名称
- 区域
- 详细地址

### 4.4 模块三：联系方式

包含字段：

- 联系人
- 联系电话
- 微信号

### 4.5 底部操作区

- 返回
- 提交注册

## 5. 字段定义

### 5.1 俱乐部封面

- 存储字段：`coverUrl`
- 类型：单图上传
- 是否必填：是
- 数量限制：1 张
- 建议比例：`16:9`
- 用途：俱乐部主页头图、品牌展示

交互说明：

- 未上传时展示上传占位
- 上传后支持预览和重新选择
- v1 不提供裁剪能力时，前端至少要限制单图并给出比例提示

### 5.2 俱乐部 Logo

- 存储字段：`logoUrl`
- 类型：单图上传
- 是否必填：是
- 数量限制：1 张
- 建议比例：`1:1`
- 用途：头像、列表卡片、活动发布主体标识

### 5.3 俱乐部名称

- 存储字段：`name`
- 类型：单行输入
- 是否必填：是
- 占位文案：`例如：企鹅羽球俱乐部`
- 规则：
  - 长度建议 2 到 30 字
  - 去除首尾空格后不能为空

说明：

- v1 前端先做非空和长度校验
- 是否允许重名由后端规则决定

### 5.4 俱乐部品类

- 存储字段：`category`
- 类型：单选 picker
- 是否必填：是
- 当前版本可选值：
  - `BADMINTON`

UI 文案：

- 羽毛球

说明：

- 当前虽然只有羽毛球，也建议保留枚举字段
- 后续扩展其他运动品类时不需要改表单结构

### 5.5 俱乐部简介

- 存储字段：`description`
- 类型：多行输入
- 是否必填：否
- 占位文案：`介绍俱乐部定位、组织频率、适合人群等`
- 字数建议：0 到 300 字

### 5.6 俱乐部地点

- 存储字段：
  - `poiName`
  - `province`
  - `city`
  - `district`
  - `address`
  - `latitude`
  - `longitude`
- 类型：腾讯地图 POI 选择
- 是否必填：是
- 交互方式：点击 `选择地图位置`

地图返回后，建议回填：

- 地点名称
- 省市区
- 详细地址
- 经纬度

说明：

- `poiName` 用于展示俱乐部地点名称
- `address` 用于展示详细地址
- 若地图返回行政区拆分不准确，允许用户手动修正展示字段

### 5.7 联系人

- 存储字段：`contactName`
- 类型：单行输入
- 是否必填：是
- 占位文案：`例如：小陈`

### 5.8 联系电话

- 存储字段：`contactPhone`
- 类型：手机号输入
- 是否必填：是
- 占位文案：`请输入联系电话`
- 规则：
  - 去除空格后不能为空
  - 需满足手机号格式校验

### 5.9 微信号

- 存储字段：`wechatId`
- 类型：单行输入
- 是否必填：否
- 占位文案：`选填，便于用户联系`

说明：

- v1 建议选填，优先降低注册门槛
- 如果后续发现联系转化依赖微信号，再升级为必填

## 6. 数据结构建议

推荐俱乐部主数据至少包含以下字段：

```ts
type ClubCategory = "BADMINTON";

type ClubStatus = "PENDING" | "ACTIVE" | "INACTIVE";

type Club = {
  id: string;
  name: string;
  logoUrl: string;
  coverUrl: string;
  category: ClubCategory;
  description: string;
  poiName: string;
  province: string;
  city: string;
  district: string;
  address: string;
  latitude: number;
  longitude: number;
  contactName: string;
  contactPhone: string;
  wechatId: string;
  status: ClubStatus;
  createdBy: string;
  createdAt: string;
  updatedAt: string;
};
```

提交成功后，系统应自动创建一条俱乐部成员关系：

```ts
type ClubMember = {
  id: string;
  clubId: string;
  userId: string;
  role: "OWNER" | "ADMIN" | "MEMBER";
};
```

默认规则：

- 当前提交用户自动成为该俱乐部 `OWNER`

与活动发布的关联关系：

- `Activity.ownerType`：`PERSONAL | CLUB`
- `Activity.ownerId`：指向 `userId` 或 `clubId`
- 俱乐部活动默认由俱乐部 `OWNER / ADMIN` 管理
- 俱乐部 `MEMBER` 仅可报名，不可代表俱乐部发布活动

## 7. 页面交互

### 7.1 初始态

页面默认展示空表单。

默认值建议：

- 品类默认选中 `羽毛球`
- 简介为空
- 微信号为空

### 7.2 图片上传

- 封面和 Logo 分开上传
- 上传成功后立即回显
- 支持替换，不支持多图
- 若上传失败，toast 提示并保留当前表单内容

### 7.3 地图选择

点击 `选择地图位置` 后：

1. 拉起腾讯地图选点或搜索地点
2. 用户完成选择
3. 返回注册页
4. 自动回填地点名称、地址、省市区、经纬度

建议文案：

- 未选择前：`请选择俱乐部地点`
- 已选择后：展示 `poiName + address`

### 7.4 提交

点击 `提交注册` 时：

1. 校验必填项
2. 校验手机号格式
3. 若通过，则提交俱乐部数据
4. 创建俱乐部成功后，自动创建当前用户为 `OWNER`
5. 按来源执行成功承接

承接规则建议：

- 若从创建活动页进入：提交成功后返回来源页，并带回新建 `clubId` 与来源标识
- 创建活动页收到返回结果后，自动选中新注册俱乐部作为发布身份
- 创建活动页中用户先前已填写的活动表单内容保持不变
- 若从“我的”页面进入：提交成功后返回“我的”或“我的俱乐部”

页面需要支持“按来源回跳 + 成功结果透传”的能力，具体路由形式可在代码中配置。

## 8. 状态设计

俱乐部建议包含以下状态：

- `PENDING`
  - 已提交，待审核
- `ACTIVE`
  - 可正常使用
- `INACTIVE`
  - 已停用，不可继续作为发布主体

v1 简化建议：

- 提交成功直接进入 `ACTIVE`

但字段层面仍建议预留 `PENDING`，避免后续接审核时重构。

## 9. 校验规则

前端至少校验以下内容：

- 封面必传
- Logo 必传
- 俱乐部名称必填
- 俱乐部名称长度合法
- 俱乐部品类必选
- 俱乐部地点必选
- 联系人必填
- 联系电话必填
- 联系电话格式正确

建议错误文案：

- `请上传俱乐部封面`
- `请上传俱乐部 Logo`
- `请填写俱乐部名称`
- `俱乐部名称长度需为 2-30 字`
- `请选择俱乐部品类`
- `请选择俱乐部地点`
- `请填写联系人`
- `请填写联系电话`
- `联系电话格式不正确`

## 10. 移动端交互建议

页面建议采用纵向卡片表单：

- 图片上传区放在最前，便于建立品牌感知
- 地图选点单独作为一张卡片，避免和纯文本输入混淆
- 联系方式放在靠后位置
- 底部按钮固定，滚动过程中始终可提交

视觉建议：

- 封面上传区域可用大卡片
- Logo 可采用圆形或圆角方形预览
- 已选中的地图地点使用信息卡展示，不直接暴露经纬度

## 11. 成功态建议

提交成功后建议 toast：

- `俱乐部注册成功`

如果后续接审核流，可改为：

- `已提交，审核通过后即可使用俱乐部身份`

## 12. v1 落地范围

本期优先实现：

1. 俱乐部注册页表单
2. 图片上传占位能力
3. 地图选点并回填地址
4. 提交创建俱乐部
5. 自动创建当前用户为俱乐部负责人

本期可暂不实现：

- 俱乐部资料编辑页
- 俱乐部详情页
- 俱乐部成员管理
- 审核后台
- 多品类扩展

## 13. 与活动发布页的边界

本 PRD 以俱乐部创建/注册为主，也包含与活动发布页之间必要的身份承接规则。

不在本篇中展开：

- 创建活动页完整 UI 结构
- 活动发布页除身份承接外的其他业务表单细节
- 俱乐部详情页、成员管理、审核后台等后续能力

如需实现创建活动页完整交互，可在活动创建页 PRD 中继续细化，但与俱乐部创建相关的身份规则以本篇为准。
